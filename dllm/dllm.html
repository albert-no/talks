What Are Discrete Diffusion Language Models?

Instead of generating text left-to-right, dLLMs start from a fully masked sequence and iteratively unmask tokens in any order.

$$x_T = [\texttt{M}]^N \;\xrightarrow{\text{denoise}}\; x_{T-1} \;\xrightarrow{\text{denoise}}\; \cdots \;\xrightarrow{\text{denoise}}\; x_0$$
  • Forward process: progressively mask tokens until fully corrupted
  • Reverse process: learn to predict masked tokens conditioned on visible ones
  • Training objective: predict each masked token given all unmasked tokens

Denoising trajectory example

[M][M][M][M][M]
↓
[M]is[M][M]blue
↓
Theis[M]notblue
↓
Theisskynotblue

Tokens can be revealed in any order — not just left-to-right

Why Do dLLMs Behave Differently?

Architectural inductive biases

  • Bidirectional attention — every token attends to every other token, enabling richer contextual reasoning
  • Weight sharing couples forward and reverse information flow
  • No causal mask → different attention patterns emerge

Training dynamics

  • Random masking at every training step → model sees all possible orderings
  • Gradient alignment between forward and reverse tasks
  • Learns order-agnostic representations

Consequence 1

Flexible generation → new control surfaces (decoding order, step count, masking strategy)

Consequence 2

Any-order generation → enlarged attack surface for safety

Consequence 3

New failure modes: EOS overflow, sensitivity to unmasking order, inter-token dependency issues

Part I

Recent Progress in the Field

From principled training to competitive LLM-scale systems

SEDD: Score Entropy Discrete Diffusion

Lou et al., ICML 2024

Key idea

Introduced score entropy — a new loss function for training discrete diffusion models that directly parameterizes the concrete score (ratio of transition probabilities).

$$\mathcal{L}_{\text{SE}} = \mathbb{E}_{t,x_0,x_t}\left[\sum_{y \neq x_t} s_\theta(y \mid x_t, t) - \log s_\theta(x_0 \mid x_t, t)\right]$$

Why it matters

  • Made discrete diffusion practically competitive with AR models at moderate scale
  • Principled connection to continuous diffusion score matching
  • Improved perplexity over prior discrete diffusion methods by large margins

Contributions

  • Novel score-entropy training objective for discrete state spaces
  • Concrete score parameterization avoids intractable normalizing constants
  • First discrete diffusion model competitive on language modeling benchmarks
  • Foundation for subsequent scaling efforts

RADD: Absorbing Diffusion ↔ Any-Order AR

Ou et al., 2024

Key insight

Reframed absorbing-state diffusion through clean-data conditional distributions and showed an equivalence to any-order autoregressive modeling.

Bridge result: The optimal denoiser for absorbing diffusion is equivalent to an any-order autoregressive model — they share the same family of conditional distributions.

Practical impact

  • Unifies two seemingly different paradigms
  • Justifies using AR-style training tricks for diffusion
  • Clean theoretical foundation for subsequent work

Absorbing diffusion

Forward: tokens → [MASK] (absorbing state)

Reverse: [MASK] → tokens (denoising)

Connection to AR

Any-order AR: $p(x_{\sigma(i)} \mid x_{\sigma(<i)})$ for permutation $\sigma$

Absorbing diffusion implicitly marginalizes over all possible orderings — same conditional family, different sampling procedure.

LLaDA: Diffusion LLMs at Scale

Nie et al., 2025

Breakthrough

First demonstration that diffusion language models can be trained at 8B parameter scale with competitive general-purpose performance.

  • Trained on large-scale text data with masked diffusion objective
  • Instruction-tuned variant (LLaDA-8B-Instruct) for chat/reasoning
  • Competitive with AR models of similar scale on diverse benchmarks

LLaDA demonstrated that the scaling laws for diffusion LLMs are favorable — performance improves predictably with compute.

Dream-Coder

Specialized diffusion for code

Applied the diffusion LLM paradigm to code generation, leveraging the natural fit between bidirectional context and structured programming.

  • Code has strong bidirectional dependencies (types, imports, function signatures)
  • Diffusion's any-order generation naturally handles fill-in-the-middle tasks
  • Competitive with AR code models on generation benchmarks

Why diffusion for code?

AR models must generate code left-to-right, but programmers don't write code that way. Diffusion models can:

  • Fill in function bodies given signatures
  • Complete imports after writing usage
  • Generate code with full bidirectional context

Rainbow Padding

The EOS Overflow Problem

Problem

Instruction-tuned dLLMs exhibit a critical failure: as allocated sequence length increases, responses paradoxically become shorter.

  • Responses collapse into streams of <eos> tokens
  • Early termination degrades output quality
  • Severity increases with generation budget
  • Affects math, code, and general tasks

Counterintuitive: giving the model more space to generate actually hurts performance

Expected

Theansweris42because...eospadpad

Actual (EOS overflow)

Theeoseoseoseoseoseoseoseos

Longer allocation → more EOS domination

Rainbow Padding

Root Cause: Dual Role of <eos>

The design flaw

Standard instruction-tuning pads variable-length sequences with <eos>, giving it a dual role:

  • Termination signal — marks end of response
  • Padding token — fills remaining positions

The cascade

  • EOS dominates later positions in training data
  • Model learns strong positional bias toward EOS
  • Confidence-based decoding selects EOS early
  • EOS propagates backward through denoising steps
  • Response collapses

Training data view

Every training example:

prompt response tokens eos eos eos eos

→ EOS appears at high frequency in later positions

→ Model conflates "end of text" with "this position"

$$p(\texttt{eos} \mid \text{position } i) \uparrow \;\text{as}\; i \to N$$

Positional bias in EOS probability causes premature termination

Rainbow Padding

Solution: Rainbow Padding

Core idea

Replace repeated <eos> padding with a cycling sequence of distinct pad tokens, reserving a single <eos> to mark the true end.

Before (standard padding)

response eos eos eos eos eos

After (Rainbow Padding)

response eos pad₁ pad₂ pad₃ pad₁ pad₂

Why it works

  • Decouples termination from padding
  • Distributes probability mass across distinct tokens
  • No single pad token dominates any position
  • Breaks the EOS feedback loop

As few as 7 distinct padding tokens are sufficient to resolve the issue

Practical deployment

  • LoRA fine-tuning for 1 epoch on minimal data
  • Works on already instruction-tuned models
  • Architecture- and dataset-agnostic

A2D

Safety Alignment for dLLMs

The enlarged attack surface

dLLMs' any-order generation creates a fundamentally different safety challenge compared to AR models.

  • Harmful spans can appear at arbitrary positions
  • Template-based prefilling attacks (e.g., DIJA) can bypass response-level refusals
  • Standard AR safety alignment techniques fail

DIJA attack success rate

>80% on baseline instruction-tuned dLLMs

Standard alignment inherited from AR models is insufficient

Why AR safety methods fail

AR safety relies on the model refusing at the beginning of a response. But in dLLMs:

  • Tokens can be generated in any order
  • Harmful content can be filled in after safe content is placed
  • Prefilling attacks fix partial responses, bypassing initial refusal

Response-level alignment is not enough — we need token-level safety

A2D

A2D: Any-Order, Any-Step Defense

Core idea

Train dLLMs to emit an [EOS] refusal signal at any masked position whenever harmful content is encountered.

$$\text{A2D}: \; p_\theta(\texttt{[EOS]} \mid x_t, \text{pos}_i) \uparrow \;\text{if harmful}$$
  • Align at token level, not response level
  • Train under randomized masking — robust to any decoding order
  • Use [EOS] probability as real-time safety monitor

Training

For harmful prompts, supervise all masked positions to predict [EOS], under various random masking patterns. This ensures robustness regardless of decoding order or step.

Inference

harmful prompt... [EOS] [EOS] [EOS]

Model refuses by emitting [EOS] at every position → safe termination

dgMARK

Watermarking for Diffusion LLMs

Motivation

As dLLMs become practical, we need attribution mechanisms — ways to determine whether text was generated by a specific model.

Challenge

  • AR watermarking relies on sequential token generation — doesn't apply to dLLMs
  • dLLMs generate tokens in arbitrary order
  • Need a new watermarking paradigm

Key observation

Practical dLLMs exhibit strong sensitivity to unmasking order, even though an ideal predictor would be invariant. This sensitivity creates a new channel for watermarking.

AR watermarking

Bias next-token distribution → detectable statistical signature

✗ Requires sequential generation

dgMARK insight

Steer unmasking order → detectable signature without reweighting probabilities

✓ Works with any-order generation

dgMARK

dgMARK: Decoding-Guided Watermarking

Method

  • Define a binary hash function that maps tokens to {0, 1}
  • At each denoising step, rank candidate positions by decoding score (confidence, entropy, or margin)
  • Steer unmasking order toward positions whose high-reward candidate tokens satisfy a parity constraint
  • Detect via elevated parity-matching statistics

Key property: dgMARK never reweights the model's learned token probabilities — it only changes which positions are unmasked first.

Plug-and-play

Compatible with existing decoding strategies:

  • Confidence-based ordering
  • Entropy-based ordering
  • Margin-based ordering
  • + One-step lookahead variant

Detection

Sliding-window detector ensures robustness under post-editing:

  • Insertion
  • Deletion
  • Substitution
  • Paraphrasing
dgMARK

dgMARK: Summary

🔑

Novel channel

Exploits unmasking-order sensitivity unique to dLLMs — a watermarking channel that doesn't exist in AR models

🔌

Plug-and-play

No retraining, no probability reweighting — works with any decoding strategy out of the box

🛡️

Robust detection

Sliding-window statistics withstand insertion, deletion, substitution, and paraphrasing

Reversal Curse

The Reversal Curse in Language Models

The phenomenon

AR models trained on "A is B" fail catastrophically on "B is A" — they cannot reverse learned factual associations.

Autoregressive models

Train: "Tom Cruise's mother is Mary Lee Pfeiffer"

Fail: "Mary Lee Pfeiffer's son is ___"

Masked diffusion models

Same training data → significantly mitigated reversal curse

Can handle both directions

Common explanation

Hypothesis: "The any-order training objective (random masking) lets the model see both directions during training"

But wait: Observing "[MASK] is B" during training does not necessarily teach the model "B is [MASK]" — these are different conditioning patterns.

Our question: If it's not just the objective, what actually explains the reversal curse mitigation in MDMs?

Reversal Curse

Key Insight: Architecture + Training Dynamics

Our answer

The mitigation arises from architectural structure and its interaction with training — not from the objective alone.

Mechanism 1

Weight sharing in transformer encoders couples forward and reverse directions

Mechanism 2

Gradient alignment — minimizing forward loss simultaneously reduces reverse loss

Both mechanisms operate through the shared attention weights in the bidirectional transformer architecture.

Intuition

In a bidirectional transformer:

Forward: A → B
Attention: A attends to B
↕ shared weights
Reverse: B → A
Attention: B attends to A

Same $W_Q, W_K, W_V$ weights participate in both directions → learning one direction implicitly helps the other.

Reversal Curse

Theoretical Analysis: Attention Score Correlation

Setup: one-layer transformer encoder

Analyze how weight sharing in $W_Q, W_K$ couples forward and reverse attention patterns.

$$\text{Forward attention: } \alpha_{A \to B} = \text{softmax}(q_A^\top k_B)$$ $$\text{Reverse attention: } \alpha_{B \to A} = \text{softmax}(q_B^\top k_A)$$

Result

Weight sharing makes forward and reverse attention scores positively correlated: when the model learns to attend strongly in one direction, it simultaneously strengthens attention in the reverse direction.

Why AR doesn't have this

In causal (decoder-only) transformers:

  • Causal mask blocks B → A attention
  • Weight sharing cannot couple directions
  • Forward training gives zero reverse signal

In bidirectional (encoder) transformers:

  • No causal mask → both directions active
  • Weight sharing creates coupling
  • Forward training implicitly helps reverse
Reversal Curse

Gradient Alignment

Second mechanism

Beyond attention coupling, we show that the gradients from forward and reverse losses are aligned.

$$\nabla_\theta \mathcal{L}_{\text{fwd}} \cdot \nabla_\theta \mathcal{L}_{\text{rev}} > 0$$

Implication

A gradient step that reduces the forward loss $\mathcal{L}_{\text{fwd}}$ also tends to reduce the reverse loss $\mathcal{L}_{\text{rev}}$. The model automatically improves on reverse queries even though it's only trained on forward data.

Gradient alignment in practice

Measured cosine similarity between forward and reverse gradients across training:

  • Consistently positive across training
  • Stronger in early layers (shared representations)
  • Explains progressive mitigation during training

This is a structural property of bidirectional transformers — not specific to any particular training objective.

Reversal Curse

Experimental Validation

Controlled experiments

  • Toy tasks: synthetic "A is B" / "B is A" reversal tasks with controlled architecture variations
  • Ablations: separate effects of objective vs. architecture
  • Large-scale: validate on full-scale diffusion language models

Large-scale validation

DAPD

The Parallel Decoding Challenge

The promise

dLLMs can unmask multiple tokens per step — enabling faster generation than sequential AR decoding.

The problem

Each denoising step only provides token-wise marginal distributions. Unmasking multiple tokens simultaneously requires accounting for inter-token dependencies.

Naive parallel decoding: unmask tokens independently → ignores dependencies → errors accumulate → quality degrades

Example

Consider: "The [M] [M] is blue"

Tokens "big" and "sky" are strongly dependent:

→ "The big sky is blue" ✓

→ "The sky big is blue" ✗

Unmasking both simultaneously without knowing their dependency can produce incoherent text.

We need to identify which tokens are weakly dependent and can safely be decoded in parallel.

DAPD

DAPD: Dependency-Aware Parallel Decoding

Core idea

Use self-attention to induce a conditional dependency graph over masked tokens, then parallelize by selecting independent sets.

  • Extract attention weights between masked positions
  • Build dependency graph: edge = strong interaction
  • Find independent set: non-adjacent nodes
  • Unmask independent tokens in parallel
  • Repeat until all tokens revealed

Training-free — no auxiliary models, no retraining. Fully plug-and-play.

Dependency graph

Edges = strong attention between masked tokens

[M]₁
[M]₂
[M]₃
[M]₄

Connected = dependent (unmask sequentially)
Disconnected = independent (unmask in parallel)

Why attention works

Self-attention naturally captures token-token interactions. High attention ≈ high conditional dependency.

DAPD

DAPD Algorithm

Algorithm: DAPD

Input: Masked sequence $x_t$, model $p_\theta$

While masked positions remain:

1. Compute attention $A = \text{Attn}(x_t)$

2. Extract masked-to-masked submatrix $A_{\mathcal{M}}$

3. Build graph $G$: edge $(i,j)$ if $A_{\mathcal{M}}[i,j] > \tau$

4. Find independent set $\mathcal{I} \subseteq \mathcal{M}$

5. Sample tokens for positions in $\mathcal{I}$

6. Update $x_t$: unmask positions in $\mathcal{I}$

Return: $x_0$

Key properties

  • Avoids co-updating strongly coupled tokens
  • Maximizes parallelism — independent set is as large as possible
  • Globally distributed updates — not just local blocks
  • Adaptive — graph changes as tokens are revealed

DAPD better exploits the any-order generation capability of dLLMs — it selects tokens that are truly safe to unmask together, regardless of position.

Key Takeaways

  • dLLMs are real. From SEDD to LLaDA 2.0, discrete diffusion language models have gone from theory to competitive 7B+ systems in under two years.
  • They behave differently. Bidirectional architecture + masked training create unique inductive biases — some beneficial (reversal curse mitigation), some dangerous (enlarged attack surface).
  • New paradigm, new control surfaces. Robustness, safety, watermarking, and decoding all require dLLM-specific solutions that exploit (or account for) any-order generation.
  • Architecture matters. The reversal curse mitigation is not from the objective — it's from weight sharing in bidirectional transformers. Understanding inductive biases is key.
  • Practical solutions exist. Rainbow Padding, A2D, dgMARK, and DAPD are all plug-and-play methods that address real deployment challenges today.
